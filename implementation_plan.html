Implementation plan · HSBC account-opening forms · v1.0

Cropping handwritten signatures from scanned forms, on CPU.

Original prompt Deep research on the latest technologies for cropping signature from a scanned file of an HSBC account opening form. Deployed to AWS ECS on x86 without GPU. What's the best solution for efficiency and accuracy? Expose as an API and MCP server. Give a concrete implementation plan with detailed specs and acceptance criteria.

A fine-tuned RT-DETRv2-S detector (Apache‑2.0), exported to ONNX with the OpenVINO Execution Provider and INT8 quantization, wrapped in a single Python service that exposes both a FastAPI REST surface and an MCP server on AWS ECS Fargate. Designed around the bank's two non-negotiables: a permissive license and no customer-data egress.

The state-of-the-art public model for this task (tech4humans/yolov8s-signature-detector, 94.7% precision / 89.7% recall) is built on Ultralytics YOLOv8, which is AGPL-3.0. Deploying it as an internal network service inside HSBC without an Ultralytics Enterprise License means AGPL's source-disclosure obligations attach to the consuming application. The plan below treats the Apache-2.0 RT-DETR path as the default and the Ultralytics path as conditional on a paid Enterprise License being on file before any AGPL weight ever runs in production.

02

Technology landscape

Three viable tiers. We pick the middle one — modern open detectors fine-tuned on the right data — and exclude the other two with reasons.

Tier A · Classical CV (excluded as primary)

Connected-component analysis + heuristics (size, ink ratio, stroke continuity) — e.g. ahmetozlu/signature_extractor. Useful as a fallback and for post-detection mask cleanup, but brittle on real bank forms: printed cursive name labels, stamps, and dense form fields fool it. keep as fallback

Tier B · Open deep detectors fine-tuned for signatures · chosen

Two public benchmarks dominate this task: Tobacco-800 and the merged signature-detection dataset by tech4humans (2,819 document images, Apache 2.0). Published numbers on that test set, ONNX Runtime on CPU:

What the tech4humans benchmark of 21 architectures taught us

  • Smaller backbones matched larger ones on this task — signatures are mid-sized objects, no need to pay for a big backbone.
  • Pure CNN one-stage detectors ran 2–4× faster on CPU than DETR variants at similar accuracy.
  • Optuna hyperparameter tuning added ~8% F1 over default training.
  • OpenVINO Execution Provider was the single largest CPU speedup lever.

Tier C · Foundation-model OCR & managed services (excluded)

PaddleOCR-VL 0.9B (Oct 2025) — overkill for this task on CPU; 50–100× slower than a 10M YOLO at no accuracy gain on detection alone.
AWS Textract SIGNATURES — pragmatic but sends customer signatures to a third-party endpoint; non-starter for HSBC data sovereignty review. Keep as an evaluation oracle only.

03

Architecture

One service, two transports, three storage hops. Everything inside one VPC; no outbound internet from the task.

Solid lines = request path. Dashed = pull-only (image fetch at task boot).

Boundary rules that fall out of this diagram

  • Task SG only accepts traffic from ALB SG. ALB SG only accepts traffic from the caller app's SG.
  • VPC endpoints for S3 / ECR / CloudWatch / Secrets Manager. No NAT egress.
  • Model weights are baked into the image. No runtime download — keeps tasks startable in air-gapped environments and removes a supply-chain risk.
  • S3 bucket is single-purpose, KMS-encrypted with a customer-managed key, 24-hour lifecycle delete on both inputs/ and outputs/.
04

Inference pipeline

What happens inside the container, from received bytes to returned crops. Single-threaded-friendly, no async-fire-and-forget on the inference call — backpressure is handled by ECS scaling, not by an internal queue.

Latency budgets are illustrative; revalidated in Phase 2 against the target instance.

Stage details

01 · Ingest. File-type detect from magic bytes. PDFs go through PyMuPDF (fitz) at 300 DPI per page; images decode with Pillow/OpenCV.

02 · Preprocess. Hough-line / projection-profile deskew (≤ ±5° correction). Cheap orientation check via a 1-line OCR snippet → rotate 0/90/180/270 if upside-down. Resize long-edge to 1024px, letterbox to 640² for the model. Optional CLAHE if mean intensity is too high or too low.

03 · Detect. Single batch, dynamic input. intra_op_num_threads pinned to the task's vCPU count. INT8 quantized weights.

04 · Postprocess. Score threshold (default 0.55, configurable per-request), class-aware NMS, map letterbox coordinates back to original page coordinates, add 8% padding (default). Optional connected-component mask within the crop to remove form-line bleed when apply_mask=true.

05 · Emit. For each signature: PNG (RGBA if mask applied), bbox JSON, confidence, page number. Either uploaded to S3 under a per-request prefix or returned base64-inline depending on return_format.

Phase 0 · Data & Legal 1–2 wks

  • Confirm with HSBC Legal that the Apache-2.0 path is acceptable; obtain a written sign-off on architecture choice.
  • Stand up labelling environment (CVAT or Label Studio) inside the VPC. Do not use hosted Roboflow for live HSBC forms.
  • Collect ~500–1,000 anonymized in-house pages. 70/15/15 split. Combine with Tobacco-800 + tech4humans datasets for pre-training breadth.
  • Define labelling guidance: single class signature; label every handwritten signature including initials; do not label printed name labels, stamps, or pre-printed flourishes.

Phase 1 · Model training 2 wks

  • Base: PekingU/rtdetr_v2_r18vd from Hugging Face (Apache 2.0).
  • HF Transformers + PyTorch. Fine-tune on a 1× T4 or A10 spot instance for a few hours.
  • Recipe: 640², AdamW, lr 1e-4 backbone / 1e-3 head, cosine schedule, 60 epochs. Augmentations: rotation ±5°, brightness/contrast ±15%, Gaussian noise, JPEG compression 50–95.
  • Optuna 20-trial sweep on dropout, lr0, box-loss weight (the tech4humans team showed this adds ~7–8% F1).
  • Gate: mAP@50 ≥ 0.92 on the held-out HSBC split.

Phase 2 · CPU optimization 1 wk

  • Export to ONNX opset 17 via HF Optimum.
  • Run onnxruntime.tools.optimizer (fusion, constant folding).
  • INT8 post-training quantization with a 200–500 image calibration set.
  • Validate accuracy degradation ≤ 1.5 mAP@50 points. If exceeded, switch to QDQ + per-channel weight quantization; if still failing, fall back to FP32.
  • Gate: p95 ≤ 350 ms on target instance with ORT + OpenVINO EP.

Phase 3 · Service implementation 1.5 wks

Single Python service; one model load shared by both transports:

Phase 4 · AWS deployment 1 wk

See §07 for the full spec. Highlights: python:3.12-slim multi-stage Dockerfile, image < 700 MB, ECS Fargate (2 vCPU / 4 GB), internal ALB, blue/green via ECS rolling update.

Phase 5 · Testing & rollout 1.5 wks

  • Unit (pre/post-processing), integration (golden image fixtures), load (Locust, 100 RPS sustained on a 4-task cluster).
  • Model card with training-data lineage, eval metrics, known failure modes, version.
  • Nightly drift monitor on a fixed canary set; alert on mAP drift > 2 points.
  • Shadow rollout against the current process for 1,000 real forms; cut over when agreement ≥ 99%.

07

MCP server

Built with FastMCP (the standard Python MCP framework). Same process, same loaded model, same internal inference.run() as the REST surface — no second container, no second model copy. Transport: Streamable HTTP behind the same ALB, path prefix /mcp/.

Tools exposed

app/mcp_server.pysketch
from fastmcp import FastMCP
from pydantic import BaseModel, Field
from .inference import run_extraction

mcp = FastMCP(name="hsbc-signature-cropper")

class ExtractParams(BaseModel):
    file_uri: str = Field(description="s3:// URI of the PDF/image")
    confidence_threshold: float = 0.55
    padding_pct: float = 0.08
    apply_mask: bool = False

@mcp.tool(
    annotations={
        "title": "Extract signatures from a document",
        "readOnlyHint": False,   # writes crops to S3
        "openWorldHint": False,  # only touches configured bucket
    }
)
async def extract_signatures(params: ExtractParams) -> dict:
    """Detect every handwritten signature on a scanned form and return
    cropped PNGs plus bounding boxes. Input must be in the approved S3
    bucket; output crops are written under a per-request prefix."""
    return await run_extraction(
        source=params.file_uri,
        threshold=params.confidence_threshold,
        padding=params.padding_pct,
        mask=params.apply_mask,
        sink="s3",
    )

@mcp.tool(annotations={"title": "Model info", "readOnlyHint": True})
def model_info() -> dict: ...

@mcp.tool(annotations={"title": "Health check", "readOnlyHint": True})
def healthz() -> dict: ...

MCP-specific safety rules

  • The tool only accepts s3:// URIs inside the pre-approved bucket prefix. No file paths, no raw bytes — forces all data through the audited storage path.
  • The tool never returns base64 crop bytes to the agent — only S3 URIs and bounding-box metadata. Signature pixels stay out of LLM context windows.
  • mask_error_details=True on the FastMCP instance so internal exception traces don't surface to the agent.
  • Per-tool RBAC via the bearer token at the ALB — only enrolled service identities can call extract_signatures.
Why expose MCP at all, when REST already exists?

The MCP surface is for agents orchestrating multi-step work — "for the twelve PDFs in this case folder, pull out the signatures and attach them to the customer record". REST is the contract for the document-processing batch system and RPA bots. Both target the same backend; they differ in who's holding the steering wheel.

AWS deployment specification

Compute

  • Fargate · x86_64 · 2 vCPU / 4 GB
  • Desired count 2, max 20
  • Target tracking: CPU 60%, ALBRequestCountPerTarget 50
  • ≥ 2 AZs, private subnets only

Networking

  • Internal ALB, target-group ip mode
  • Health check /v1/readyz, healthy threshold 2
  • SGs: ALB ← caller-app SG; task ← ALB SG only
  • VPC endpoints: S3, ECR, CloudWatch, Secrets Manager
  • No NAT egress

Storage

  • ECR private repo, image scanning on
  • S3 hsbc-sig-cropper-{env}
  • SSE-KMS, customer-managed key
  • Lifecycle: delete inputs/ and outputs/ after 24 h
  • Block Public Access: all four toggles on
  • Bucket policy: task role only

Secrets & config

  • Thresholds, bucket names, KMS key IDs, model version pins via Secrets Manager / SSM
  • Model ONNX is baked into the image — update = new image tag = blue/green

Observability

  • Structured JSON logs to CloudWatch. Zero payload bytes, request_id propagated.
  • CloudWatch Metrics: latency, success rate, detections-per-page, p95/p99 latency.
  • Alarms: p95 > 800 ms, error rate > 1%.
  • X-Ray traces if downstream services are already X-Ray enabled.

9.3 · API & MCP contract

  • OpenAPI 3.1 spec published; CI validates implementation matches spec.
  • MCP server passes mcp-inspector connectivity test on all three tools.
  • Idempotent on request_id within 24 h.

9.4 · Security & compliance

  • No signature pixel data in any log line. Verified by CI regex check against test-run log output.
  • TLS 1.2+ only at the ALB; PFS ciphers only.
  • trivy image scan: zero CRITICAL CVEs at deploy time. Monthly rescan with alerts on new CRITICAL/HIGH.
  • IAM least privilege validated against documented permissions via iam-policy-simulator.
  • Data retention: inputs and outputs deleted from S3 within 24 h, verified by automated inventory check.
  • Audit log captures: timestamp · request_id · caller identity · page count · signature count · model version. Never signature bytes.

9.5 · Operational

  • Two ECS tasks survive a single-AZ failure (chaos test: stop one task, p95 returns to baseline within 60 s).
  • Image is reproducible: docker build from tagged source produces byte-identical image (multi-stage, pinned deps, --no-cache-dir).
  • Runbook covers: model rollback, threshold tuning, dependency CVE patch, drift-alert response.
  • Model card and OpenAPI spec checked into the repo, version-aligned with the image tag.

9.6 · License & legal

  • Final architecture is Apache-2.0 end-to-end (training code, inference code, weights), or a signed Ultralytics Enterprise License is on file before any AGPL weight runs on the production cluster.
  • pip-licenses inventory generated, reviewed, and stored alongside the model card.
  • CI gates on pip-licenses --fail-on AGPL — accidental AGPL pull-in is build-blocking.

Risks & mitigations

Risk Likelihood Impact Mitigation
AGPL surprise — a transitive dep pulls in AGPL code under our app's runtime med critical Automated license scan (pip-licenses --fail-on AGPL) gating CI; quarterly manual review of requirements.txt
Customer data leakage via logs or LLM context (MCP) low critical Hard rule: pixels never enter logs or LLM context. MCP returns S3 URIs only. CI regex check on log output. Bucket policy denies any role outside the task role.

Week-1 checklist

Five concrete items. Once items 3 and 4 land, the remaining work is fine-tuning, hardening, and rollout — all well-understood, predictable engineering.

  1. 01 Stand up the labelling environment inside the VPC. Export the tech4humans Apache-2.0 dataset as a starting set.
  2. 02 Get 50–100 anonymized HSBC pages from a sympathetic stakeholder for a smoke-test set.
  3. 03 Spike: clone PekingU/rtdetr_v2_r18vd, fine-tune 20 epochs on a T4 spot, export to ONNX, benchmark on a c7i.large EC2. Sanity-check that we land near published numbers and that latency on the target hardware is acceptable.
  4. 04 In parallel: scaffold the FastAPI + FastMCP service skeleton with a stub model that returns canned detections. Deploy to a dev ECS Fargate service. Confirm the whole rail (ALB → ECS → S3 → logs → readyz) works end-to-end before the real model lands.
  5. 05 Legal sign-off on the Apache-2.0 path so it can't block the launch later.

Open questions

Do we support multi-signature ordering, or just unordered crops?

Some downstream systems care about "applicant signature" vs "witness signature". The detector returns positions but not roles. Cheapest path: pair detection with a layout-anchor lookup ("witness" label nearest each bbox). More robust: a tiny classifier head on the crop.

Decide with · product + downstream integration owner, before phase 3
Async endpoint: SNS push, or polling only?

Polling is simpler and one fewer IAM coupling. SNS is nicer for the consuming app. Default to polling for v1, add SNS only if a real consumer asks.

Decide with · platform integrations, before phase 4
Retention of the audit S3 copy

The processing copy is 24-hour TTL. A separate audit-trail bucket with longer retention (e.g. 7 years for KYC) may be required. This is a Legal/Compliance call, not an engineering one — but it changes the bucket layout in §08.

Decide with · HSBC compliance, before phase 0 exit
