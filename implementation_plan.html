Ship the published best-in-class detector. No training.

What changed from v1

v1 assumed we'd fine-tune on a custom HSBC corpus. v2 takes the published model as-is and ships. The deltas are concentrated in three places: phase plan, license, and acceptance criteria (we now measure the model's fit rather than train it to a target).

v1 · with training
  • Phase 0 · Data labelling (1–2 wks)
  • Phase 1 · Model training + Optuna (2 wks)
  • Phase 2 · INT8 quantization + ONNX export (1 wk)
  • Phase 3 · Service implementation
  • Phase 4 · AWS deployment
  • Phase 5 · Testing & rollout
  • 500–1,000 in-house labelled samples
  • T4/A10 spot instance for training
  • Bespoke model card
v2 · published model
  • Phase 0 · Legal & procurement (1 wk)
  • Phase 1 · Evaluation set + smoke test (1 wk)
  • Phase 2 · Service implementation (1 wk)
  • Phase 3 · AWS deployment (3–5 days)
  • Phase 4 · Shadow rollout & cutover (1 wk)
  • ~ 100-page HSBC eval set only
  • No training hardware required
  • Inherit tech4humans' model card + ONNX exports

The architecture, API contract, MCP server, and AWS deployment from v1 are all preserved. What changed is everything upstream of "we have an ONNX file on disk" — that file is now a download, not a deliverable.

Same training set, same evaluation protocol. Slight accuracy hit (~1 mAP@50 point), ~3× slower per page on CPU. Loaded via Hugging Face Transformers (AutoModelForObjectDetection). Ship-ready with no procurement friction.

design choice Build for both — config-switch at deploy

The two models have different input shapes and output formats, but the wrapping is small. We expose MODEL_BACKEND={yolov8s|cdetr50} as an env var; the rest of the service is identical. This lets us ship the Apache-2.0 path on day one and swap to YOLOv8s the moment Legal signs off on the Enterprise License, with zero service-code change.

Why these two and nothing else

  • They're the only two pretrained models on HF (or anywhere public) that have been benchmarked specifically against the signature-detection task with published numbers. Generic object detectors fine-tuned on COCO don't transfer well; signature-specific training is the difference.
  • The author (tech4humans) is the same — they published a paper, a dataset, both model variants, and an inference server. Single point of provenance for audit.
  • Both already export to ONNX and have OpenVINO-EP handlers in the model repo (handler.py). The hard work for CPU deployment is already done by the upstream author.

03

Architecture

Unchanged from v1. One service, two transports, three storage hops. Inside one VPC; no internet egress from tasks.

Solid lines = request path. Dashed = pull-only (image fetch at task boot). ONNX baked into image; no runtime download.

Boundary rules

  • Task SG only accepts traffic from ALB SG. ALB SG only accepts traffic from the caller app's SG.
  • VPC endpoints for S3 / ECR / CloudWatch / Secrets Manager. No NAT egress.
  • The downloaded ONNX file is baked into the Docker image at build time, pulled from Hugging Face in CI, then never fetched at runtime. Image checksum is the model checksum.
  • S3 bucket is single-purpose, KMS-encrypted with a customer-managed key, 24-hour lifecycle delete on both inputs/ and outputs/.

Inference pipeline

Latency budgets revised against the published YOLOv8s figure (172 ms per page on ONNX-CPU). With our preprocessing and serialization overhead added, typical end-to-end is ~230 ms.

Numbers based on tech4humans' published ONNX-CPU benchmark for YOLOv8s. Conditional-DETR adds ~330 ms to stage 3.

What we inherit from the upstream repo

  • yolov8s.onnx — exported, ready to load with onnxruntime.
  • handler.py — reference inference handler using ORT + OpenVINO Execution Provider; we adapt the pre/post-processing from this.
  • Hugging Face Space (Gradio demo) — useful for smoke testing without writing any code.
  • Reference inference server tech4ai/t4ai-signature-detect-server — Triton + OpenVINO Docker setup, optional reference for production.

What we still write

  • PDF → page rasterization (PyMuPDF) — upstream only handles single images.
  • Deskew + orientation check — upstream doesn't include these.
  • Crop + padding + optional mask cleanup — upstream returns bboxes only.
  • FastAPI + FastMCP wrapping — upstream serves Triton, which we replace with a thinner single-process server.

Phase 0 · Legal & procurement 1 wk · parallel

  • Legal review on the two paths — Ultralytics Enterprise License (primary, ~$5k–50k/yr, scope-dependent) vs Apache-2.0 Conditional-DETR (fallback, no procurement).
  • Initiate Enterprise License procurement via Ultralytics' contact form — they quote per use case; a bank with HSBC's scope should expect mid-five-figures annually.
  • Build a small "decision memo" with both paths and let Legal pick. Engineering work continues against the Apache-2.0 path so we're never blocked.

Phase 1 · Evaluation set + smoke test 1 wk · parallel

  • Collect ~100 anonymized HSBC account-opening pages spanning the form variants you actually see in production.
  • Label them once, by hand, in CVAT or Label Studio (in-VPC) — single class signature. Should take 1 person about a day.
  • Run both models against this set via a notebook, not yet the service. Output: a 3-row table of precision / recall / mAP@50 / latency for {YOLOv8s, Conditional-DETR}. This is the moment of truth.
  • Gate: at least one model achieves recall ≥ 0.90 at precision ≥ 0.95 on the HSBC set. If not, see Risks §10.

Phase 2 · Service implementation 1 wk

Single Python service. Model factory pattern so both backends are switchable:

Model download happens at build time, not runtime:

Dockerfile · build-time model fetchmulti-stage
FROM python:3.12-slim AS model-fetch
ARG MODEL_BACKEND=yolov8s
ARG HF_TOKEN

      huggingface-cli download tech4humans/yolov8s-signature-detector \

      huggingface-cli download tech4humans/conditional-detr-50-signature-detector \
        --local-dir /models ; \
    fi

FROM python:3.12-slim
COPY --from=model-fetch /models /opt/models
COPY requirements.txt /tmp/
RUN pip install --no-cache-dir -r /tmp/requirements.txt
COPY app/ /app/
ENV MODEL_BACKEND=$MODEL_BACKEND  MODEL_PATH=/opt/models
EXPOSE 8080 8081
CMD ["python", "-m", "app.main"]

Phase 3 · AWS deployment 3–5 days

Spec unchanged from v1 — see §08. Highlights: python:3.12-slim base, image < 700 MB, Fargate 2 vCPU / 4 GB, internal ALB, blue/green via ECS rolling update. The only addition: image tag now includes the model backend (e.g. signature-cropper:1.0.0-yolov8s) so a backend swap is a deployment swap.

Phase 4 · Shadow rollout + cutover 1 wk

  • Run the new service in parallel with whatever the current process is. Diff outputs on 1,000 real forms.
  • Pick the operating confidence threshold based on the Phase 1 + shadow data.
  • Cut over when agreement ≥ 99% on the shadow set. Keep the old path warm for one week post-cutover.
  • Nightly drift-monitor job from day one: run the model on a fixed canary set, alert on metric drift > 2 points.

MCP server

Unchanged from v1. Same process, same loaded model, same internal inference.run(). The MCP transport is Streamable HTTP behind the same ALB, path prefix /mcp/.

app/mcp_server.pysketch
from fastmcp import FastMCP
from pydantic import BaseModel, Field
from .inference import run_extraction, model_info_payload

mcp = FastMCP(name="hsbc-signature-cropper", mask_error_details=True)

class ExtractParams(BaseModel):
    file_uri: str = Field(description="s3:// URI of the PDF/image")
    confidence_threshold: float = 0.50
    padding_pct: float = 0.08
    apply_mask: bool = False

@mcp.tool(
    annotations={
        "title": "Extract signatures from a document",
        "readOnlyHint": False,   # writes crops to S3
        "openWorldHint": False,  # only touches configured bucket
    }
)
async def extract_signatures(params: ExtractParams) -> dict:
    """Detect every handwritten signature on a scanned form and return
    cropped PNGs plus bounding boxes. Input must be in the approved S3
    bucket; output crops are written under a per-request prefix.

    Returns S3 URIs only — never raw pixel bytes to the agent."""
    return await run_extraction(
        source=params.file_uri,
        threshold=params.confidence_threshold,
        padding=params.padding_pct,
        mask=params.apply_mask,
        sink="s3",
    )

@mcp.tool(annotations={"title": "Model info", "readOnlyHint": True})
def model_info() -> dict:
    """Returns the active backend, HF revision SHA, and last eval metrics
    on the HSBC eval set."""
    return model_info_payload()

MCP-specific safety rules

  • The tool only accepts s3:// URIs inside the pre-approved bucket prefix. No file paths, no raw bytes.
  • The tool never returns base64 crop bytes to the agent — S3 URIs and bounding-box metadata only.
  • mask_error_details=True so internal exception traces don't surface to the agent.
  • Per-tool RBAC via the bearer token at the ALB.

AWS deployment specification

Identical to v1 except the image tag now encodes the model backend.

Compute

  • Fargate · x86_64 · 2 vCPU / 4 GB
  • Desired count 2, max 20
  • Target tracking: CPU 60%, ALBRequestCountPerTarget 50
  • ≥ 2 AZs, private subnets only

Networking

  • Internal ALB, target-group ip mode
  • Health check /v1/readyz, healthy threshold 2
  • SGs: ALB ← caller-app SG; task ← ALB SG only
  • VPC endpoints: S3, ECR, CloudWatch, Secrets Manager
  • No NAT egress

Storage

  • ECR private repo, image scanning on
  • Image tag pattern: signature-cropper:{ver}-{backend}
  • S3 hsbc-sig-cropper-{env}
  • SSE-KMS, customer-managed key
  • Lifecycle: delete inputs/ and outputs/ after 24 h
  • Block Public Access: all four toggles on

Secrets & config

  • Thresholds, bucket names, KMS key IDs, MODEL_BACKEND via Secrets Manager / SSM
  • Model ONNX baked into image at build time — no runtime fetch

Observability

  • Structured JSON logs to CloudWatch. Zero payload bytes, request_id propagated.
  • CloudWatch Metrics: latency, success rate, detections-per-page, p95/p99 latency, plus model_backend dimension so we can compare YOLOv8s vs Conditional-DETR if both ever run.
  • Alarms: p95 > 600 ms (YOLOv8s) / > 1200 ms (Conditional-DETR), error rate > 1%.

Acceptance criteria

The semantics shift in v2: thresholds describe what the published model must actually achieve on HSBC data, not what we trained it to. If the model can't hit these on the Phase 1 eval set, that's a decision point, not a remediation.

9.1 · Accuracy on HSBC eval set ~100 pages, hand-labelled in Phase 1

MetricYOLOv8s targetConditional-DETR target
Precision @ tuned threshold≥ 0.93≥ 0.91
Recall @ tuned threshold≥ 0.88≥ 0.86
mAP@50≥ 0.92≥ 0.90
False-positive rate on blank-signature pages (20 fixtures)≤ 3%≤ 3%
Median bbox IoU vs. human label (TP only)≥ 0.80≥ 0.80

Targets are ~2 mAP points below published numbers to allow for the domain shift between Tobacco-800 and HSBC's form distribution. If we beat these, great. If we miss them by more than 5 points, see Risks §10.

9.3 · API & MCP contract

  • OpenAPI 3.1 spec published, CI-validated against the implementation.
  • MCP server passes mcp-inspector on all tools.
  • Idempotent on request_id within 24 h.
  • Response includes model_version and model_backend for audit.

9.4 · Security & compliance

  • No signature pixel data in any log line. CI regex check on test-run log output.
  • TLS 1.2+ only at the ALB; PFS ciphers only.
  • trivy image scan: zero CRITICAL CVEs at deploy time.
  • IAM least privilege validated against documented permissions.
  • Data retention: inputs and outputs deleted from S3 within 24 h.
  • Audit log captures: timestamp · request_id · caller identity · page count · signature count · model_backend · HF revision SHA.

9.5 · Operational

  • Two ECS tasks survive a single-AZ failure (chaos test).
  • Image is reproducible: docker build from a tagged commit produces byte-identical image (pinned deps, --no-cache-dir, HF revision pinned).
  • Runbook covers: model rollback, threshold tuning, dependency CVE patch, drift-alert response, switching MODEL_BACKEND.

9.6 · License & legal

  • If shipping with YOLOv8s: signed Ultralytics Enterprise License on file before the first production deploy.
  • If shipping with Conditional-DETR: Apache-2.0; no procurement needed.
  • pip-licenses inventory generated and reviewed.
  • CI gates on pip-licenses --fail-on AGPL when MODEL_BACKEND=cdetr50 (defensive: shouldn't be any AGPL deps in the Apache-2.0 path).

Risks & mitigations

v2's biggest risk is concentrated in one place: domain shift between the upstream training distribution (Tobacco-800 + signatures-xc8up) and HSBC's forms. Everything else is operational.

RiskLikelihoodImpactMitigation
Domain shift — published numbers don't translate to HSBC forms (different layout, logo, ink colors, scanner stack) med high Phase 1 eval set tells us the real number before any AWS commit. If the model misses the 9.1 thresholds by more than 5 points, we re-add a fine-tuning phase as a v3 effort. If by less, accept and ship.
Enterprise License procurement stalls med low Apache-2.0 path (Conditional-DETR) is always shippable. Engineering work proceeds against the Apache path so we're never blocked on procurement. Backend swap is one env var.
Customer data leakage via logs or LLM context (MCP) low critical Hard rule: pixels never enter logs or LLM context. MCP returns S3 URIs only. CI regex check on log output. Bucket policy denies any role outside the task role.
Upstream HF model is taken down or moved low med Model ONNX baked into image at build time (Phase 2 Dockerfile pattern). Once built and pushed to ECR, the upstream availability is irrelevant. Image is the artifact of record.
Form layouts change — HSBC redesigns the account-opening form med med Nightly drift monitor on a fixed canary set. Quarterly re-eval. If drift sustained, that's the trigger to do a v3 fine-tune.
Model misses signatures in unusual ink colors (red, faded) med med Tobacco-800 is mostly black-ink; the published model has limited red-ink exposure. Specifically test this in the Phase 1 eval set. Mitigations if it fails: per-channel grayscale-pre-conversion before inference, or a v3 fine-tune.
Printed cursive signatures (sample watermarks, signed name labels) cause false positives med low Phase 1 eval set should include negative fixtures. If FP rate too high, post-process filter on bbox aspect ratio + stroke density.
ONNX Runtime / OpenVINO EP regression on a new release low med Pin onnxruntime-openvino version. Validate accuracy + latency on every dependency bump in CI.

Week-1 checklist

Six concrete items. The first three run in parallel.

  1. 01 Procurement: submit the Ultralytics Enterprise License request via their contact form. Provide deployment scope (internal-only, ~4 Fargate tasks, ~1M pages/month). Expect 1–4 week turnaround.
  2. 02 Eval set: get 100 anonymized HSBC account-opening pages from a sympathetic stakeholder. Label them once by hand in CVAT/Label Studio (in-VPC). 1 person-day.
  3. 03 Smoke test: in a notebook, download both tech4humans/yolov8s-signature-detector and tech4humans/conditional-detr-50-signature-detector from Hugging Face, run them against the eval set, produce the precision/recall/mAP table from §09.1. This is the go/no-go data point.
  4. 04 Service scaffold: spin up the FastAPI + FastMCP skeleton from §05 Phase 2 against a stub model that returns canned detections. Deploy to a dev ECS Fargate service. Confirm ALB → ECS → S3 → logs → readyz all wire up before the real model lands.
  5. 05 Legal memo: one-page comparison of the two paths for Legal. Cover license (AGPL vs Apache), accuracy delta (~1 mAP@50 point), latency delta (~3× slower for fallback), and the option to ship Apache today and swap to YOLOv8s when the Enterprise License lands.
  6. 06 Eval review: at end of week 1, walk the team through the §09.1 numbers from item 03. If we're above target, go. If we're between target and "5 points below", go with caveats. If more than 5 points below, escalate — that's a v3 conversation about adding training back.

Open questions

Ship Apache-2.0 first and swap, or wait for the Enterprise License?

Shipping Apache-2.0 first unblocks production weeks earlier and validates the entire pipeline end-to-end. Switching the backend later is a deploy, not a redesign. Recommended.

Decide with · engineering + product, end of Phase 1
Do we keep both models warm in production for fallback?

Loading both adds ~200 MB to image size and ~150 MB to RAM but means a model failure on one is auto-handled by the other. Probably not worth the operational complexity for v1.

Decide with · SRE, before Phase 3
If domain shift is real, where does the fine-tune budget come from?

If Phase 1 shows the published model can't hit 9.1 thresholds on HSBC data, we have an honest conversation: accept the lower accuracy with a manual-review fallback, or fund a v3 fine-tune phase (adds ~3–4 weeks). Decision is much easier after we have the Phase 1 numbers.
